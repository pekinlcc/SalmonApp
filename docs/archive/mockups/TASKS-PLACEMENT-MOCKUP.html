✦ Tasks/待办 应该放在哪里？

背景：Pulse 给的 suggestedAction 里包含 task 步骤（例 "提交报名表 5/11 前"）。 Google 那边有 Tasks 服务、Outlook 有 To Do —— 都能云端同步。问题是 UI 上放哪。下面 4 个方案。

A日历视图加 "待办" 标签页 我推荐

在 CalendarView 顶部加两个标签：📅 日程 / 📋 待办。同一个视图，两种数据。 日程仍走 Google Calendar / Graph Events，待办走 Google Tasks / Graph Todo。共用账号 OAuth。

优点
  • 分类符合直觉：时间约束的东西都在"日历"下
  • 账号共用 OAuth，加一个 scope 就行（https://www.googleapis.com/auth/tasks）
  • 侧栏不增加新项，UI 不臃肿
  • Pulse 的 task 步骤可以直接创建到这里 + toast 提示
缺点
  • "任务" 和 "事件" 在概念上有差别 — 一些用户可能想要明确区分
  • 切 tab 略多一步

C日历里直接混排（不分 tab）

待办作为一种特殊的"事件"显示在周视图里。带 due 日期的任务出现在当天，没 due 的放在一个"无期限"折叠区。

优点
  • 一目了然：进 SalmonApp 第一眼就能看到 待办 + 今日聚焦
缺点
  • 首页越来越胖。混排原则是"AI 已经帮你按重要性排了" — 多一个清单反而稀释聚焦
  • 没有时间维度，多于 5-6 个待办时首页会很拥挤
我的建议是 A — 日历视图加 "📋 待办" tab。理由：
  1. 有 due date 的待办本质上是"未来某天的待办事项" — 跟日历同源
  2. 侧栏不增项，整体克制
  3. 实现时复用 CalendarView 现有的账号选择 + 同步按钮基础设施
  4. Pulse 的 task 步骤一键创建 → toast + Google Tasks / Graph Todo 云端同步

你选哪个？A / B / C / D，或者你有别的想法？决定之后我立刻实现，下次 build 你确认了再装。

实现时需要扩 OAuth scope（Google: tasks；Microsoft: Tasks.ReadWrite）， Gmail 用户要重新走一次 OAuth（点 "＋ 添加 Gmail" 重新登录）。
